# GraphLink Communication Playbook

## Core Shift
Stop thinking like a builder (correctness).
Start thinking like a communicator (reaction).

Your goal is not to explain.
Your goal is to make developers **feel a problem instantly**.

---

## 1. Don’t Explain — Provoke

Bad:
> “GraphLink is a GraphQL code generation tool”

Good:
> “Java GraphQL clients are broken—and here’s why”

Focus on:
- pain
- frustration
- strong opinions

---

## 2. Content Formula (Always Follow This)

Every post/video should follow:

1. **Pain (specific)**
   - “You’re writing `TypeReference<...>` every time”

2. **Agitation**
   - “That’s not type safety—it’s noise”

3. **Contrast**
   - Show ugly vs clean

4. **Minimal solution**
   - One command → result

If one part is missing → content weakens.

---

## 3. Target One Audience First

❌ Too broad:
- Java + Dart + Flutter + full-stack

✅ Focus:
- Spring Boot Java developers

Speak directly to:
- their tools
- their frustrations
- their patterns

---

## 4. Volume Over Perfection

Don’t aim for perfect content.

Aim for:
- 10–20 posts/videos quickly
- test what gets reactions

Track:
- comments
- saves
- clicks to GitHub

Most content will fail. That’s expected.

---

## 5. First 5 Content Ideas

1. **Why Java GraphQL clients feel wrong**
2. **What a clean GraphQL call should look like**
3. **Before vs After: Spring Boot GraphQL**
4. **Stop sending your entire schema**
5. **I built this because I was annoyed**

---

## 6. Demo Rule (20-Second Understanding)

Your showcase must instantly answer:

- What problem?
- What changes?
- What do I gain?

Avoid:
- deep architecture
- long explanations

---

## 7. Harsh Truth

Good engineering ≠ adoption.

If you don’t communicate clearly:
> nobody will use it

---

## 8. Golden Rule

Before posting, ask:

> “Would this make a developer stop scrolling?”

If not → rewrite the hook.

---

## 9. What to Emphasize

Push these hard:

- No generics in Java GraphQL calls
- No runtime dependency
- Minimal queries (no schema spam)
- Schema = single source of truth

---

## 10. What to Avoid

- Generic “tool introduction”
- Over-explaining architecture
- Trying to please all audiences
- Long, dense first impressions

---

## Final Mindset

You’re not building a tool anymore.

You’re:
- identifying pain
- amplifying it
- showing a cleaner alternative

Do this consistently for a few weeks and you’ll start seeing traction.
